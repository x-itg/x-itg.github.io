批量修改，先问该不该用脚本

在直接改、先分析、生成脚本之间切换模式，才是真正省 Token 的工程判断

itg · 公众号: 火星火箭 · 嵌入式AI工程化系列

我做 AI 辅助开发这么久，攒了不少省钱的小技巧。其中最管用、也最容易被滥用的一招是：批量操作，别急着让 Agent 直接改代码。先判断该不该生成脚本。

很多人一看到“几十个文件一起改”，第一反应就是让 Agent 直接动手。它当然能改，但问题不在“能不能改”，而在你是不是选对了执行模式。模式选错了，白烧 Token；模式选对了，一轮对话能省下后面三四轮返工。

两种模式，选对就是省钱

模式一：直接改

Agent 逐个文件分析，逐个修改。每一步都是一轮对话，每一步都在消耗 Token。改漏了，把错误贴回去再改；改错了，再来一轮。

模式二：生成脚本

Agent 输出一个可执行的 Python 脚本。你审阅脚本，运行，一次完成所有修改。脚本同时输出变更日志；有问题，改脚本重跑，而不是逐个文件返工。

两种模式各有适用场景。选错模式，要么白花钱，要么白花时间。选对模式，省下的不只是这一轮对话，而是后续整串来回。

怎么选？两条规则就够了

规则一：可复用的，或者重复修改量大的，生成脚本

什么叫“可复用”？下次遇到类似任务，改两个参数就能继续用。什么叫“重复修改量大”？同一类修改模式在几十个文件里反复出现。

这种场景用脚本，省的不只是这一次的 Token。生成一次，复用十次，每次省三轮对话，十次就是三十轮。脚本本身还是一份决策记录：改了哪些文件、改了什么、什么时候改的，全都可以落进日志。

规则二：重复量大但不可复用的，让 Agent 先给建议，再决定怎么改

这类任务需要人的判断介入。最优流程不是直接让 Agent 动手，而是先让它分析影响范围、列出修改方案、指出风险。你确认方案后，再决定让它直接改，还是生成一个一次性脚本。

这中间多了一步人工审阅，但这正是省钱的关键。方案审对了，后面省下的不止是 Token，还有返工的时间。方案审错了，跑完才发现方向不对，再回头改，那才是真金白银打水漂。

两个条件都不满足，就直接改。 修改量小、不可复用，直接让 Agent 改完审阅，最快。为这种任务写脚本，等于用牛刀杀鸡，牛刀本身还费油。

为什么直接改容易费钱

很多人习惯让 Agent 直接操作文件。这其实是把 AI 当成了一个“会改代码的编辑器”，但它不是编辑器，它是按轮次计费的。

Agent 直接改代码，交互模式天然就是反复来回。让它改一个文件，它改了；跑一遍测试，发现两个地方漏了；把错误贴回去，它再改；改完再跑，又引入一个新问题；你再补上下文，它再动手。每一步都在烧 Token。

如果遇到复杂的批量修改，来回三四轮是常事，而且完全没有复现性。下次遇到同样的任务，还是得从头再来一遍。这种模式最大的浪费，不是某一次输出长了，而是每一轮都在重新建立同一份上下文。

生成脚本，省在哪

让 Agent 生成脚本，交互模式就完全变了。Agent 不是直接改代码，而是输出一个可执行、带注释的脚本。你审阅一遍，通过，运行。脚本做完所有修改，同时输出一份完整的变更日志。

如果运行结果有问题，把变更日志和错误信息贴回给 Agent，Agent 改脚本，你再运行。这个循环里，每一轮都在同一个脚本文件上迭代，而不是在几十个分散的源文件上来回跳转。

还有个关键细节：让 Agent 生成脚本时，同时生成一份恢复脚本。万一批量改完发现方向不对，一键恢复，回到原点。这和闭环收敛是一回事：有回滚机制兜底，人才敢放手让系统自己跑。

一个该用脚本的例子

项目需要把所有源文件里的函数名 old_func 替换成 new_func，同步更新头文件引用，最后跑一遍测试。

这个任务完美符合“可复用且重复量大”的标准。如果直接改：让它改第一处，发现头文件没同步，再让它补；补完跑测试，挂了，把错误贴回去；它说还有三处漏了，再改。三四轮对话，Token 哗哗地烧。

如果生成脚本：一轮对话，Agent 输出脚本和恢复脚本。审阅通过，运行，不到十秒，整个项目改完，变更日志清清楚楚。直接改要三四轮，生成脚本只要一轮，这就是模式选择的差价。

一个不该用脚本的例子

还是替换函数名，但加了条件：有三处是故意保留的兼容逻辑，不能改，其他的都要改。

这个任务不满足“可复用”的标准。规则需要逐例判断：哪三处是兼容逻辑？为什么不能改？脚本无法自动做这个判断。硬让 Agent 生成脚本，它大概率会无差别全部替换，然后在注释里写一句“请手动检查并恢复”。

正确做法是让 Agent 先给建议：把包含 old_func 的文件逐一列出来，对每个文件给出“该不该改”的分析。你来确认，Agent 来执行。虽然多花了分析那一轮，但保证了正确性。在这种需要逐例判断的任务上，省下来的 Token 不够填出错的坑。

一句话总结

批量修改，先判断：可复用或重复量大的，生成脚本；重复量大但不可复用的，先让 Agent 给建议，再决定怎么改；两个都不满足的，直接改。

生成脚本时，别忘了让 Agent 同时输出一份恢复脚本，留好回滚的退路。省 Token 的核心不是“用脚本”，而是用对模式。模式选对了，省下的不止是 Token，是返工的时间、出错的风险，以及下次不必从头再来的重复劳动。

法则不是死规则，是活的判断框架。 在正确的场景选择正确的工具，这个判断力本身，才是整套方法论里最值钱的部分。
