别让AI画图，让它写图

SVG是代码，不是图片——用编号精准控制每一个元件，用分层策略拆解每一张复杂框图

嵌入式AI工程化系列 · 工程心法

核心结论

AI画不好位图，但它天生就该用代码画图。SVG的本质不是图片，是几十行、几百行用XML写的矢量指令。而写代码、改代码，恰恰是AI最擅长的事。用编号精准定位每个元件，用分层策略从小图到大图逐级构建，这就是工程思维在设计领域的投影。

I · 核心理念

一、SVG是代码，不是图片

我做设计有一个很深的体会：AI画不好位图，但它天生就该用代码画图。这是两种完全不同的工作方式。你让AI去编辑一张PNG图片——调整某个图标的位置、修改某根连接线的粗细、换掉某个颜色——它做得非常吃力。因为位图是像素矩阵，改一个细节意味着重新理解整张图的视觉布局，这对目前的AI来说负担很重，效果也很不稳定。

但SVG不同。SVG的本质不是图片，是代码。是几十行、几百行用XML写的矢量指令。打个比方：SVG 就像一份"烹饪食谱"——它不是一张做好的菜的照片，而是写清楚了"放多少盐、炒几分钟、摆盘方式"的操作步骤。食谱可以随意修改某一步，照片只能推倒重拍。

位图是像素矩阵，改一个细节就要重新理解整张图。SVG是结构化代码，每一根线、每一个色块背后都是清晰的语法。AI不需要"看懂"这张图，它只需要"读懂"这段代码。

二、用编号精准定位每个元件

我在实践中摸索出一个非常有效的技巧，就是给SVG里的每个元素打上编号。让AI或Agent生成整张框图的初稿时，要求它给每一个视觉元素——发射机、交换机、连接线、文字标注——都分配一个唯一的编号。这些编号在SVG代码里以注释的形式存在，在视觉呈现上就是标注在每个元件旁边的数字。

这个编号系统有什么用？它把人和AI之间的沟通成本降到了最低。

以前没有编号的时候，我需要这样告诉AI："把左边第三个发射机的颜色改深一点，把右上角那根连接线的箭头改大一点，把右下角那个标注的字体加粗。"这种描述模糊、不精确，AI理解起来很吃力，改错位置的概率很高。

有了编号，沟通变成这样："把元件12的颜色改深一点，把连接线34的箭头改大一点，把标注56的字体加粗。"AI不需要理解"左边第三个"是哪个，它只需要找到代码里编号为12的元件，直接修改。精准、高效、一步到位。

编号系统的工作流

生成初稿时要求AI给每个元件编号 → 视觉呈现上数字标注在元件旁边 → 修改时直接用编号精准定位 → 所有细节调完后，让AI删除全部编号 → 输出最终成品。编号是施工期间的脚手架，交付时不需要留在成品里。

❌ 没有编号

"把左边第三个发射机的颜色改深一点"——模糊、不精确，AI改错位置概率高

✅ 有编号

"把元件12的颜色改深一点"——精准、高效、一步到位，沟通成本降到最低

三、从小图到大图的分层策略

如果框图规模比较小——十几个元件，几根连接线——直接让DeepSeek在网页上生成整张SVG就够了。它一次性输出完整的代码，调整一两轮就能定稿。

但如果框图规模很大——几十个元件，复杂的连接关系，多层级的结构——就需要用到工程里的Agent来进行集成布局。流程是这样的：

  1. 先用DeepSeek生成每一个独立的SVG元件，全部带上编号。
  2. 再把所有编号过的元件交给Agent，告诉它整体的框架结构、各个元件的大致位置、连线关系。
  3. Agent负责把这些元件精确地摆放到页面上，画好连接线，统一配色和标注。
  4. 最后人在Agent输出的整张图上进行细节微调——通过编号精准定位，告诉Agent改哪里。

分层策略的核心

复杂问题拆成多个简单步骤，每一步都用最合适的工具去解决。这和在工程里拆解复杂系统的逻辑完全一致——拆解与整合讲的就是这件事，只不过这里拆解的不是代码模块，而是视觉元件。

🟢 小图 · 直接出

十几个元件、几根连接线——DeepSeek网页版直接生成整张SVG，一两轮定稿

🟡 大图 · 分层来

几十个元件、复杂连接、多层结构——先生成独立元件，再用Agent集成布局

极简示例：三步搭一张大图

假设要画一张"嵌入式系统通信架构图"，包含 3 个传感器、2 个控制器、1 个上位机，总共 20+ 个元件。分层做法如下：

第 1 层：元件原子（DeepSeek 生成）

每个元件单独生成 SVG，带编号：

<!-- #1 温度传感器 -->
<g id="sensor-temp">
  <rect x="10" y="10" width="80" height="40"
        rx="6" fill="#27ae60"/>
  <text x="50" y="35" text-anchor="middle">温度</text>
</g>

<!-- #2 湿度传感器 -->
<g id="sensor-humi">
  <rect ... /> ...
</g>

第 2 层：组装布局（Agent 集成）

把所有元件交给 Agent，用 <use href="#sensor-temp" x="100" y="50"/> 精确定位，自动画连接线。

第 3 层：人工微调（编号定位）

看到成品后，告诉 AI："把 #3 号连接线改成虚线，把 #7 号元件颜色改成蓝色。" 改完后删除编号，输出成品。

整个过程就像拼乐高：先做零件，再拼总成，最后微调细节。

四、这才是真正发挥了AI的优势

AI的强项不是理解像素，是理解代码。SVG的每一根线、每一个色块、每一段文字，背后都是清晰的语法结构。AI可以在语法层面精准地创建、修改、删除。它不需要"看懂"这张图，它只需要"读懂"这段代码。

而人在这个流程里的角色，不是亲自画图，是定义风格、审核输出、做只有人才能做的审美判断。AI负责生成和修改，人负责定义和裁决。

这套工作流的核心优势在于精准和高效。AI对SVG代码的操作是像素级的精确，每一个坐标、每一个颜色值、每一个线宽，都可以通过修改代码精准控制。没有模糊地带，没有视觉偏差。而且一次建模，反复复用。SVG元件一旦生成，就进了素材库。下次再做类似的框图，不需要重新生成，直接从库里拿出来，让Agent重新布局就行。一套元件，反复复用，迭代成本趋近于零。

人AI分工法则

AI擅长批量，人擅长判断。AI对SVG代码的操作是像素级的精确。人只在最关键的细节上介入——通过编号精准定位，告诉AI哪个元件需要调整、哪个连接线需要修改。两者各做各擅长的事，效率自然最高。

这和够用原则完全一致。工具该做什么就让它做什么，人不该做的就不要做。AI擅长代码操作，就让它在代码层面做所有能做的事。人擅长审美判断和细节把控，就只在最关键的节点介入。不是用最贵的模型，不是用最复杂的工具，而是用最合适的方式，在最合适的节点上施加杠杆。

与传统 AI 绘图方案的核心差异

很多人会问：Midjourney、DALL·E 这些工具不也能画图吗？为什么还要绕一圈用 SVG + MCP？答案在于交付物的本质不同：

维度Midjourney / DALL·ESVG + MCP 工具链
交付物像素位图（PNG/JPG）结构化 XML（.drawio / .svg）
可编辑性无法局部修改，只能重新生成按编号/ID 精准修改任意元件
版本管理无法 diff，只能存多张图纳入 Git，每行变更清晰可追溯
复用性每次从零生成元件一次建模，反复复用
动画支持无CSS @keyframes / flowAnimation
核心理念AI 渲染像素AI 写代码，工具渲染

一句话总结：传统 AI 画图像"约稿"，SVG + MCP 像"施工"。前者交付的是成品图，后者交付的是可继续编辑、可版本管理、可纳入 CI/CD 的工程文件。

II · 实战工具链

五、从风格蓝图到可编辑工程图：MCP 工具链实战

前面四节讲的是理念：SVG是代码、编号精准定位、分层拆解、人AI分工。这一节进入实战——把理念落地为一条完整的工具链：DeepSeek 定风格 → GitHub Copilot Agent Mode 调工具 → Draw.io MCP 生成可编辑工程图。

核心理念：AI不渲染，只写XML

很多AI画图方案让AI直接生成图片，结果是静态截图，无法二次编辑、无法纳入版本管理。

Next AI Draw.io 的设计思路相反：AI不负责渲染，只负责生成和修改Draw.io的底层XML（XML 就是一种结构化的数据格式，类似于填表格——每个字段都有固定位置，改一个字段不影响其他内容）。浏览器里的Draw.io嵌入组件负责展示，AI产出的是一份可继续编辑、可导出、可版本管理的工程文件。

打个比方：传统方案像让AI当"画家"直接在画布上涂油彩；而这里AI是"建筑师"，它只写施工图纸（XML），渲染交给专业的绘图软件。图纸可以反复修改，而油画只能推倒重来。

MCP Server 的架构清晰说明了这一点：

┌─────────────────┐ stdio ┌─────────────────┐
│ AI Agent        │<──────│ MCP Server      │
│ (Copilot/Claude)│       │ (this package)  │
└─────────────────┘       └────────┬────────┘
                                   │
                           ┌───────▼────────┐
                           │ Embedded HTTP  │
                           │ Server (:6002) │
                           └───────┬────────┘
                                   │
                           ┌───────▼────────┐
                           │ User's Browser │
                           │ (draw.io embed)│
                           └────────────────┘
通信原理一句话：Copilot 通过 stdio（标准输入输出，类似于两个人通过对讲机对话）向 MCP Server 发送工具调用请求，MCP Server 内部通过 HTTP 转发给浏览器中嵌入的 draw.io，再把操作结果原路返回。整条链路无需联网，全部在本地完成。

AI 职责三层拆分

在此基础上，进一步拆分 AI 职责：

  • DeepSeek：生成符合风格要求的 SVG 代码（颜色、线条、动画），作为"设计蓝图"
  • GitHub Copilot Agent Mode：通过 MCP 协议调用 Draw.io 原子工具，将风格蓝图转化为 Draw.io XML
  • CodeGraph：将项目文件索引为本地知识图谱，AI 查询图谱而非逐文件扫描

这验证了一条更完整的链路：风格定义（DeepSeek）→ Agent 工具调用（Copilot + MCP）→ 结构化资产（.drawio + .svg）→ 本地知识图谱（CodeGraph，零成本反复查询）。

环境准备：VS Code + MCP 配置

基础环境要求：VS Code 最新版、Node.js v18+（Windows 建议 v20+）、GitHub Copilot 扩展（1.5.x+，确保 Agent Mode 功能完整可用）。

Node.js 版本兼容性说明： v18.x：MCP Server 可正常运行，但 CodeGraph 要求 ≥ 20。如果只用 MCP 不用 CodeGraph，v18 够用。 v20.x：推荐版本，MCP + CodeGraph 全部功能可用，npx 缓存机制更稳定。 v22.x：最新 LTS，功能完全兼容，但部分旧版 npm 包可能触发 deprecation 警告，不影响使用。 建议用 nvm（Windows 用 nvm-windows）管理多版本，随时切换。

VS Code 通过配置文件集成 MCP 服务器——在 .vscode/mcp.json 中定义，VS Code 自动管理服务器生命周期。步骤：

  1. 在项目根目录创建 .vscode 文件夹
  2. 创建 mcp.json 并粘贴以下内容：
{
  "mcpServers": {
    "drawio": {
      "command": "npx",
      "args": ["@next-ai-drawio/mcp-server@0.2.0"]
    }
  }
}
生产环境提示：建议固定版本号（如 @0.2.0），避免 latest 标签变化带来的影响。如果用内网环境，可通过环境变量 DRAWIO_BASE_URL 指定自托管 draw.io 地址。

确保在 Copilot Chat 面板中选择 Agent Mode。Agent Mode 支持调用 MCP 工具完成多步骤任务。如果模式切换入口不明显，可以点击 Copilot Chat 面板顶部的模式选择器下拉菜单找到"Agent"选项。

风格先行：用 DeepSeek 生成 SVG "风格原子"

在调用 MCP 之前，先让 DeepSeek 定义图表的"视觉风格"——这是传统 AI 画图方案缺失的一环：风格与结构分离。

任务示例：生成 LSTM 架构图中"动画连接线"的 SVG 元素代码。在 DeepSeek 对话中输入：

"请生成一段 SVG 代码，表示 LSTM 单元中从'遗忘门'到'细胞状态'的连接线。需要虚线、箭头、流动动画（CSS @keyframes），颜色为橙色。同时给出该连接线所附着节点的位置示意（x,y 坐标）。"

DeepSeek 返回类似这样的 SVG 片段：

<defs>
  <marker id="arrowOrange" markerWidth="10" markerHeight="10"
          refX="9" refY="3" orient="auto">
    <path d="M0,0 L0,6 L9,3 z" fill="#FF8C00"/>
  </marker>
  <style>
    @keyframes dashFlow {

MCP 五个核心原子工具

工具用途关键参数
start_session打开浏览器并建立实时预览无（必须最先调用）
create_new_diagram用 XML 新建图xml
edit_diagram按 cell ID 增删改操作描述
get_diagram读取当前图的 XML无
export_diagram保存为 .drawio 文件filepath

实测流程：三步走通

  1. 启动预览：在 Copilot Chat（Agent Mode）中说"调用 drawio 的 start_session 工具，打开 draw.io 浏览器预览窗口"。MCP Server 启动本地 HTTP 服务器（默认端口 6002），浏览器自动打开 draw.io 嵌入窗口。⚠️ start_session 必须先调用，否则后续操作报"没有活跃会话"。端口被占用时自动尝试下一个可用端口。
  2. 生成架构图：粘贴 DeepSeek 的风格蓝图，让 Copilot 调用 create_new_diagram 创建完整 LSTM 图，包含输入门（绿色）、遗忘门（橙色）、候选门（蓝色）、输出门（紫色）、细胞状态和隐藏状态。所有连接线带 flowAnimation=1——这是 draw.io 官方支持的连接器动画样式参数。
  3. 导出资产：确认预览符合预期后，调用 export_diagram 导出为 .drawio 源文件，可在 Draw.io 桌面版或网页版继续手动编辑。

交付的不是截图，是工程文件

最终得到一个 .drawio 源文件——可继续编辑、可导出 PNG/SVG、可纳入 Git LFS 做版本管理。把 DeepSeek 生成的优质 SVG 风格片段保存为"风格原子"库，未来直接复用。

六、CodeGraph：让 AI 改图时不再从头扫描

上面解决了"AI 画图"的单次成本。但真正消耗 Token（可以理解为 AI 的"阅读字数"，读得越多费用越高）的是后续修改。

每次对 AI 说"把输入门的颜色改成蓝色"，AI 都需要重新扫描文件来定位目标节点。这个成本会随着图表规模递增。CodeGraph 解决了这个问题。

原理：预索引的本地知识图谱

CodeGraph 是一个本地代码知识图谱工具（类似于给你的代码库建了一本"电话簿"，AI 查号码而不是挨家挨户敲门），通过 MCP 协议为 AI 编程助手提供预索引的语义信息。

核心原理很简单：提前把代码库中的文件结构、符号关系、调用路径全部解析成 SQLite 数据库。AI 直接查询图谱获取所需信息，而不是通过 grep、glob、read 反复扫描文件。一次索引，无限复用。

官方实测数据

CodeGraph 在 7 个真实开源代码库（覆盖 7 种语言）上进行了基准测试。测试方式：对比 AI Agent 在有和没有 CodeGraph 的情况下回答架构问题的效果，取中位数如下：

指标平均节省
成本（费用）35%
Token 消耗59%
响应时间49%
工具调用次数70%

在 VS Code 仓库（约 10,000 个 TypeScript 文件）上，Token 节省高达 73%，工具调用减少 72%。规律：收益随代码库规模递增——大仓库上代理通过几次索引查询即可给出答案，零文件读取；小型项目上收益边际收窄。

成本节省（35%）小于 Token 节省（59%），原因是 CodeGraph 的 instructions 本身会常驻 context，且 Claude 的 prompt caching 有折扣。但工具调用次数减少 70% 是实打实的效率提升。

  1. 零安装运行（推荐）：交互式安装程序自动检测已安装的编程智能体，自动配置 MCP 服务器——支持 Claude Code、Cursor、Codex CLI、GitHub Copilot Agent Mode 等。

  1. 重启 GitHub Copilot：重启 VS Code 或重新加载 Copilot 面板，MCP 服务器自动加载。之后只要项目里有 .codegraph/ 目录，Copilot 就会自动使用 CodeGraph 的查询工具。
  2. 验证生效：在 Copilot Chat（Agent Mode）中输入"使用 codegraph_explore 查看当前项目的文件结构"，如果配置成功，AI 会调用 CodeGraph 的工具而非逐文件扫描。

在这个工作流中具体省在哪里？

  • 修改图表前，AI 调用 codegraph_search 直接定位目标节点，无需 read 整个 .drawio 文件
  • 数百节点的图，CodeGraph 一次返回所有受影响的边，不用遍历
  • .drawio 修改后保存，索引自动增量更新，下次查询拿到最新状态

III · 生产与排查

七、生产环境接入注意事项

  1. 先调用 start_session：否则所有操作报"没有活跃会话"。建议在自动化脚本中加入会话健康检查逻辑。
  2. 确保 Agent Mode 启用：普通 Chat 模式无法调用 MCP 工具，务必在 Copilot Chat 面板中选择 Agent Mode。
  3. 端口冲突处理：MCP Server 默认使用 6002 端口启动嵌入式 HTTP Server。如果被占用，它会自动尝试下一个可用端口，无需额外配置。
  4. 私有化部署配置：内网环境可通过环境变量 DRAWIO_BASE_URL 指定自托管 draw.io 地址。
  5. 固定 MCP 版本：生产使用 @next-ai-drawio/mcp-server@x.y.z，避免 latest 漂移。当前稳定版本是 0.2.0。
  6. CodeGraph 的 Node 版本要求：需要 Node.js ≥ 20.0.0。如果版本过低请先升级。
  7. CodeGraph 的收益随项目规模递增：大型项目（千级以上文件）收益最明显，小型项目收益边际收窄。
  8. 风格蓝图可沉淀为模板库：把 DeepSeek 生成的优质 SVG 风格片段保存为"风格原子"，未来直接复用。
  9. 制品保存：将 .drawio 纳入 Git LFS 或制品仓库，可回滚、可 diff。

IV · 收束

九、结语：这条路已经走通了

GitHub Copilot Agent Mode 与 MCP 的集成，使得结构化的图表生成任务可以嵌入日常 AI 驱动的编程工作流。DeepSeek 进一步拆解了 AI 的职责——DeepSeek 负责"画风"，Copilot 负责"画骨"。最终交付的不是一张截图，而是一个可继续编辑、可导出、可纳入 CI/CD 的工程文件。

再加上 CodeGraph，这个工作流才真正走完"省钱闭环"：画图的 Token 花在该花的地方，改图的 Token 通过一次索引无限复用。

从"AI写SVG代码"到"DeepSeek定风格 + Copilot调MCP工具"再到"CodeGraph省钱闭环"，这条链路验证了同一个内核：

风格定义（DeepSeek）→ Agent工具调用（Copilot + MCP）→ 结构化资产（.drawio + .svg）→ 本地知识图谱（CodeGraph）

AI不渲染像素，只操作结构化XML。人不做机械搬运，只负责方向、边界和审美裁决。

如果你也在使用 GitHub Copilot Agent Mode，这个工具链值得亲自试一次。最好的验证方式不是看文章，而是直接动手：

动手验证步骤

先用 DeepSeek 生成一套 SVG 连接线元素，然后在 VS Code 中配置好 draw.io MCP，让 Copilot Agent Mode 画出带动画连接器的 LSTM 架构图。再把项目纳入 CodeGraph 索引，让 AI 修改图表——看它第二次是否不再从头扫描文件。

如果最后你得到一个可以继续编辑的 .drawio 文件，并且 Copilot 在修改时通过 CodeGraph 图谱查询完成——那就说明，从"风格描述"到"可编排工程图"，再到"可维护工程图"的完整链路，已经走通了。这不是在设计，这是在用工程思维重新定义设计。君子不器讲人和工具的关系，这篇讲人和AI在视觉创作里的关系。同一套法则，同一个内核。
