第零层 · 起点

Vibe Coding 的爽与痛

Vibe Coding 让生成速度第一次压倒了人类手写，但它在嵌入式里格外脆弱。你可以五分钟得到一个驱动框架，却很难在第六分钟确认它是不是在真实硬件上按你的时序活着。

实时性约束

中断、超时、时钟树和总线时序不会因为代码看起来“像对的”就放你过去。

可观测性缺失

没有结构化日志，AI 只能猜你现在死在初始化、状态机、还是硬件边界。

上下文割裂

单文件修改无法表达协议、工程目录、测试脚本和烧录链路之间的依赖。

反馈周期过长

只要还停留在“人手动编译、烧录、点按键、复制日志”，AI 就不可能真正进入闭环。

第一性结论：AI 的问题不是“会不会写代码”，而是“能不能看见代码在系统里有没有真的生效”。
第一招 · 双向可观测性

结构化日志 + 负日志

传统日志只记录“发生了什么”，但大量嵌入式问题的根因其实是“应该发生却没有发生”。负日志就是把这种缺失显式写进系统里。

#define LOG_STATE(state, epoch) \
    LOG("INFO", "state=%s epoch=%d", state_to_str(state), epoch)

#define EXPECT_WITHIN(ms, event_name, condition) do { \
    uint32_t start = get_tick_ms(); \
    while (!(condition)) { \
        if ((get_tick_ms() - start) > (ms)) { \
            LOG("NEGATIVE", "%s expected '%s' within %dms", __func__, event_name, ms); \
            break; \
        } \
    } \
} while(0)

显式声明期望

从“我怀疑没有收到 COMMIT”变成“系统明确声明 50ms 内必须收到 COMMIT”。

暴露静默失败

中断没触发、状态没迁移、epoch 不一致，这些过去最难抓的缺失会被主动抛出来。

让 AI 直接诊断

AI 不再从几十行杂乱 printf 里猜，而是从一条负日志开始逆推根因。

正日志记录事实，负日志记录缺失的事实。两者叠在一起，AI 才真正拥有“看见系统”的能力。
第二招 · 全局视角

统一工程上下文

AI 不是在读一个文件，而是在读一个工程。只有把固件、上位机、文档、日志和规则一起摆在它面前，它才可能做出稳定判断。

project/
├── firmware/                # 固件源代码
├── host/                    # CLI 调试与测试工具
├── docs/                    # 协议、硬件、接线和规范
├── logs/                    # 运行日志与异常样本
├── .vscode/tasks.json       # 一键自动化命令
└── agents.md                # AI 协作规约

文档也是上下文

协议说明和硬件连接图不是附录，而是状态判断和修复建议的边界条件。

日志资产化

典型日志和异常日志变成可复用样本，让 AI 逐步拥有项目记忆。

agents.md 约束行为

把“先查负日志”“先映射状态码”“先跑健康检查”写成规约，而不是靠临场提醒。

让目录表达结构

目录本身就是低成本知识图谱，决定了 AI 能否快速理解谁依赖谁。

上下文一旦统一，AI 的诊断质量不再靠运气，而开始靠结构稳定输出。
第三招 · 分工协作

多模型协作：低成本写，高性能审

不是所有任务都要最强模型。真正省钱的方式不是“全程便宜”或“全程最强”，而是让不同模型只干自己最擅长的部分。

1

模板生成交给低成本模型

驱动骨架、CLI 框架、任务清单和目录约束都属于高重复结构化产出。

2

Diff 审查和根因分析交给高性能模型

深度推理场景才值得付更高单价，比如理解负日志链路、状态机冲突和修复取舍。

3

日志预处理再次交给低成本模型

先把长日志压缩成关键事件和缺失事件，再交给高性能模型做最终判断。

第四招 · CLI 闭环

让 AI 也拥有“执行权”

只要测试链路仍然依赖人手操作，AI 就只是顾问。CLI 的作用不是优雅，而是把执行权交还给自动化。

设备与总线

python host/cli_can.py --send 0x123 01020304
python host/cli_can.py --listen --timeout 5
python host/cli_can.py --ping

日志与命令

python host/cli_serial.py COM3 115200
python host/cli_serial.py COM3 115200 "show state"
{
  "label": "full_loop",
  "dependsOn": ["build", "flash", "monitor_log", "cli_test"],
  "group": { "kind": "test", "isDefault": true }
}
当 build、flash、monitor 和 cli_test 都能命令化，AI 就不再停在“建议修改”，而能走到“执行回归”。
第五招 · 完整闭环工程

Vibe+Verify：把前四招固化成工程飞轮

前四招解决的是单点能力，第五招解决的是“如何把它们长期保持为一套可复制工程资产”。

1. 契约生成

模板生成必须满足：
1. 固件包含结构化日志和负日志
2. host/ 下具备可调用 CLI
3. tasks.json 可一键 full_loop
4. agents.md 写清状态映射和行为准则
5. 输出完整可运行工程，而不是碎片代码

2. 一键 full_loop

$ make full_loop
[BUILD] build completed
[FLASH] device updated
[LOG] collecting logs/current.log
[TEST] ANNOUNCE -> COMMIT
[FAIL] expected 'COMMIT' within 50ms
[ACTION] AI reads log, proposes diff, reruns loop

3. 人机共情

把状态码映射、硬件名称、已知陷阱和约束写进 agents.md，让“共情”从个人习惯升级成团队契约。

4. 飞轮效应

案例库

每修复一次问题，就沉淀一次“负日志模式 + 修复 diff”的经验样本。

低成本前置过滤

越多结构化经验进入案例库，后续问题就越接近低成本匹配而不是高成本重新推理。

行业概念这里的升级关键差异
传统日志结构化正日志不只写“到了哪”，还写状态、epoch、关键参数。
断言负日志不是程序崩掉式判断，而是对“缺失事件”的持续记录。
Debug LoopCLI 自动闭环从人工回归变成 AI 可执行回归。
System Promptagents.md从一次性提示升级成项目内可版本化契约。
最终目标不是“让 AI 写得更快”，而是“让 AI 进入一个能自证、能回归、能积累记忆的工程系统”。
