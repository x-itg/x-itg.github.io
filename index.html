AI编程调试的五层进化

第零层 · 起点

Vibe Coding 的爽与痛

2025年初，「Vibe Coding」火了 —— 你只要对着 AI 描述想法，它就能直接生成能跑的代码。这让从0到1变得极快：5分钟得到 CAN 驱动框架，10分钟写个测试脚本…… 但是，当项目复杂度上升，问题就来了：

  • 没有日志 -> AI 只能猜测“可能是终端电阻，可能是波特率”，给你列一张排查清单。
  • 只看片段代码 -> AI 不了解你的完整状态机，给出的建议常常离谱。
  • 手工编译烧录 -> 改一行代码等几分钟，心流断裂。
根本问题：AI 没有“眼睛”，看不到运行时状态。我们必须为它构造一套工程化的可观测性系统。
第一招 · 双向可观测性 原创

结构化日志 + 负日志：让AI同时看见“发生了”和“没发生”

传统日志只能记录“代码执行到了哪里”。但大量隐蔽Bug的根源是某件应该发生的事却没有发生（中断未触发、状态未迁移、超时未复位）。负日志正是为此而生：显式声明期望事件，若超时未到达，系统自动输出一条负日志。

1. 结构化日志回顾（正日志）

#define LOG(level, fmt, ...) \
    printf("[%s] %s:%d " fmt "\r\n", level, __FILE__, __LINE__, ##__VA_ARGS__)

LOG("INFO", "state=%s epoch=%d", state_to_str(state), epoch);
// 输出: [INFO] can_slave.c:412 state=ONLINE_FIXED epoch=2

2. 负日志：期望契约引擎

实现一个极轻量的期望槽数组，适合资源受限MCU：

typedef struct {
    const char *event_name;
    uint32_t timeout_ms;
    uint32_t deadline_ticks;
    uint8_t occurred;
    uint16_t context_state;
    uint8_t context_epoch;
} NegativeExpect;

static NegativeExpect exp_table[4];

#define EXPECT_WITHIN(ms, event_str, ...) \
    do { \
        int idx = find_free_expect(); \
        if (idx >= 0) { \
            exp_table[idx].event_name = event_str; \
            exp_table[idx].timeout_ms = ms; \
            exp_table[idx].deadline_ticks = get_tick() + ms_to_tick(ms); \
            exp_table[idx].occurred = 0; \
            exp_table[idx].context_state = current_state; \
            exp_table[idx].context_epoch = current_epoch; \
        } \
    } while(0)

void check_negative_logs(void) {
    for (int i = 0; i < MAX_EXPECT; i++) {
        if (!exp_table[i].occurred && get_tick() > exp_table[i].deadline_ticks) {
            LOG("NEGATIVE", "expected '%s' within %dms, not occurred. state=%d epoch=%d",
                exp_table[i].event_name, exp_table[i].timeout_ms,
                exp_table[i].context_state, exp_table[i].context_epoch);
        }
    }
}

3. 真实案例：CAN自组织地址协议

void send_announce(void) {
    can_send(ANNOUNCE_FRAME);
    EXPECT_WITHIN(50, "COMMIT_frame", current_state, epoch);
}

void can_rx_callback(can_frame_t *frame) {
    if (frame->id == COMMIT_ID) {
        EVENT_OCCURRED("COMMIT_frame");
    }
}

若对方无响应，超时后自动输出负日志：

[NEGATIVE] can_slave.c:234 expected 'COMMIT_frame' within 50ms, not occurred. state=CLAIMING epoch=1

AI读取后无需猜测，直接明确：“COMMIT帧未在预期窗口内到达”。结合同时刻其他日志，可以立刻锁定 epoch 隔离失效或仲裁逻辑错误。

创新价值：负日志将工程师脑中的“隐性期望”转化为运行时断言，补全了可观测性的最后一块拼图。

第四招 · CLI闭环测试 + VSCode一体化

用命令行工具打造可编程的自动化测试环境

传统调试依赖GUI烧录工具、串口助手、CAN分析仪界面，每次手动点击。真正的闭环必须所有操作都能通过命令行调用，从而集成到脚本和VSCode任务中。

1. 构建CLI测试工具链

用Python包装USB-CAN、USB串口和Socket通信，提供统一的命令行接口：

# host/cli_can.py - 使用python-can库操作USB-CAN适配器
import can
import argparse

def send_can_frame(channel, id, data):
    bus = can.interface.Bus(channel=channel, bustype='socketcan')  # Linux
    msg = can.Message(arbitration_id=id, data=data, is_extended=False)
    bus.send(msg)

def listen_can(channel, timeout):
    bus = can.interface.Bus(channel=channel, bustype='socketcan')
    msg = bus.recv(timeout)
    if msg:
        print(f"[CAN] ID={msg.arbitration_id} DATA={msg.data.hex()}")

if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("--send", nargs=2, help="--send  ")
    parser.add_argument("--listen", type=float, help="--listen ")
    args = parser.parse_args()
    if args.send:
        send_can_frame("can0", int(args.send[0]), bytes.fromhex(args.send[1]))
    if args.listen:
        listen_can("can0", args.listen)

import serial, sys, time

def send_and_read(port, baud, cmd, timeout):

2. 集成自动化测试脚本

编写一个综合测试脚本 host/test_automation.py，能够自动完成：发送特定CAN帧 → 读取串口日志 → 检查是否出现预期日志或负日志 → 返回测试结果。

#!/usr/bin/env python3
# host/test_automation.py
import subprocess, sys, re

def test_epoch_sync():
    # 1. 发送触发帧
    subprocess.run(["python", "cli_can.py", "--send", "0x123", "01020304"])
    # 2. 捕获串口日志（假设已通过 tasks.json 中的 monitor 保存到文件）
    with open("/tmp/serial.log", "r") as f:
        logs = f.read()
    # 3. 检查正日志和负日志
    if re.search(r"epoch mismatch", logs):
        print("FAIL: epoch mismatch detected")
        return 1
    if re.search(r"NEGATIVE.*expected 'COMMIT_frame'", logs):
        print("FAIL: COMMIT frame missing")
        return 1
    print("PASS: epoch sync ok")
    return 0

if __name__ == "__main__":
    sys.exit(test_epoch_sync())

            "problemMatcher": []

现在，按下 Ctrl+Shift+B → full_loop，系统自动完成：编译 → 烧录 → 启动串口日志收集 → 运行自动化测试脚本 → 输出PASS/FAIL。整个流程完全无需人工干预，AI也可以直接解析测试脚本的输出。

CLI闭环的核心价值：所有工具（USB-CAN、串口、Socket）都可通过命令行调用，意味着它们可以被集成到自动化脚本、CI/CD流水线、以及AI Agent的自主操作中。这是迈向全自动调试的基石。
💡 实战经验：将USB-CAN适配器的操作封装成CLI后，你可以在AI的指导下直接运行 python cli_can.py --send 0x200 0100 来模拟对手节点，再观察负日志。这种“脚本化测试 + 日志分析”的闭环，让AI能够像人类工程师一样主动设计实验。

共情 · 被忽略的维度

对 AI 的“共情”才是高手的终极武器

AI不知道 state=6 是 ONLINE_FIXED，也不知道你的USB-CAN设备路径。共情就是主动填补这些信息：提供枚举映射表、设备说明、以及CLI工具的用法示例。编写 agents.md 可以约束AI的行为，让它主动询问缺失信息。

把“理所当然”的常识写成AI能读懂的文档，包括CLI命令的预期输出格式。共情做得越好，AI给出的调试建议越精准。

进阶锦囊：让成本再降一半，效率翻倍

异步批量生成 —— 异步API生成多个模板变体，价格再降50%

调试案例库 —— 保存日志特征+diff，新问题直接复用
agents.md 规范 —— 统一AI行为，要求优先分析负日志和CLI测试输出

让 AI 成为真正的伙伴

从那次失败的CAN调试，到结构化日志、负日志，再到CLI自动化闭环，我一步步构建了这套“让设备说话、让AI看懂、让测试脚本化”的工程体系。现在，我每次新项目都从Vibe生成模板开始，然后一键进入full_loop，AI同时分析正日志和负日志，并通过CLI自动执行回归测试。
