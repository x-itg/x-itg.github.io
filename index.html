📡 信道入侵 —— 日志即真相

那是我第一次尝试让 AI 辅助调试 CAN 自组织地址协议。我把代码贴给模型，描述了“节点 A 收不到 COMMIT”的现象。AI 认真地给出了五六种可能原因：终端电阻没接、波特率误差、ID 滤波器配置错误……我一一排查，却毫无进展。直到我将一段串口日志随手复制进对话框，AI 忽然说：“等等，你的日志里打印了 epoch=1 但对方节点发送的是 epoch=2。这就是问题——epoch 隔离机制没有生效，所以帧被忽略了。”

那一刻我意识到：我给了 AI 足够的信息，但一直忽略了最重要的“眼睛”——日志。从那以后，我总结出一套“让设备说话，让 AI 看懂”的闭环方法。下面就是这套方法的核心技巧，每个都来自真实的踩坑与顿悟。

🔧 技巧一：结构化日志 + 闭环调试 —— 让设备“说话”，让 AI “看见”

◆ 问题 传统 printf("state=%d", state) 输出的数字需要人工翻译，AI 更是一头雾水。且调试周期长，常常依赖猜测。

◆ 解决方案 结构化日志 + 闭环反馈：

  • 输出语义化名称：state=ONLINE_FIXED 而非 state=6
  • 包含变量名：epoch=3 final_addr=0x05
  • 固定键值对 + 文件名行号 (__FILE__:__LINE__) → VSCode 终端超链接
  • 闭环流：AI写代码 → 编译烧录 → 设备输出结构化日志 → 点击跳转源码 → 复制日志给AI → AI分析 → 修改 → 重试
#define LOG(level, fmt, ...) \
    printf("[%s] %s:%d " fmt "\r\n", level, __FILE__, __LINE__, ##__VA_ARGS__)

LOG("INFO", "commit guard waiting uid=0x%08X announce=%d commit=%d", uid, announce, commit);
// 输出: [INFO] pwr_can_slave.c:412 commit guard waiting uid=0x4724E08D announce=1 commit=0

✅ 效果：AI 能直接读懂状态；点击日志瞬间跳转源码；每次修改都有真实运行数据支撑，不再盲猜。

📡 技巧二：上下位机统一工程上下文

将固件(C)、上位机(Python)、协议文档、典型日志放在同一目录，整体交付给 AI。

project/
├── firmware/          # STM32 固件 (src, inc, Makefile)
├── host/              # PC 端测试脚本 (can_test.py)
├── docs/              # protocol.md
└── logs/              # normal_run.txt

🚀 效果：AI 能追踪完整数据流（PC命令 → CAN帧 → MCU状态机 → 应答帧），消除信息断层。

🤖 技巧三：多模型协作（侧边栏执行 + Web端审核）

IDE 侧边栏 AI（Copilot/Continue）负责快速生成、重构；Web端大模型（DeepSeek/千问/GPT）负责代码审查、日志分析、架构建议。分工协作，优势互补，成本可控。

🎙️ 技巧四：语音转文字描述需求

用语音输入代替键盘，直接“说出”复杂逻辑，转文字后粘贴给 AI，灵感流畅不中断。特别适合边调试边记录思路。

⚙️ 技巧五：统一集成开发环境（VSCode + CLI工具链）

尽量使用终端可调用的调试工具（USB-CAN CLI、串口命令行、Python脚本），将所有命令集成到 VSCode 的 tasks.json，实现一键编译、烧录、测试、日志监控，所有输出集中在终端，鼠标点击即可跳转源码。

// .vscode/tasks.json 示例片段
{
    "label": "build", "command": "make",
    "label": "flash", "command": "openocd -f interface/stlink.cfg ...",
    "label": "test_can", "command": "python host/can_test.py --port COM3"
}

🧠 技巧六：保存成功案例，形成知识库

将成功的对话记录、最终代码、问题分析整理成项目知识库，后续会话直接提供给 AI，大幅减少重复解释，加速迭代。

📦 技巧七：使用版本控制记录 AI 修改历程

每次 AI 修改后提交 Git，并把 git diff 给 AI 看，让它可以基于历史变更持续优化，避免每次都从零开始。

🌌 结尾：回到那次失败的自组织调试，以及更深的领悟

那次偶然的日志粘贴让 AI 瞬间定位 epoch 不匹配。后来我又遇到 AI 给出错误建议，我追问原因，它说：“因为我误以为 state=6 是错误状态，实际却是 ONLINE_FIXED。” 那一刻我醒悟：与 AI 协作，不只需要技术闭环，还需要“共情”——理解 AI 的局限，主动把“工程师默认知道”的常识翻译成 AI 能读懂的文本，比如提供状态名映射表、硬件特殊行为说明。

共情，不只是对人的同理心，也是对 AI 的“心智模拟”。当你站在 AI 的视角思考“它缺什么信息？可能误解什么？”你就会自然优化日志格式、补充上下文。这，正是闭环调试的灵魂。

现在，每次调试复杂 bug，我先让设备输出结构化日志，然后连同上下位机代码、术语表一起交给 AI。AI 不再是盲目聊天的模型，而是能亲眼观察现场、给出精准建议的资深工程师。

这套方法同样适用于后端服务、PC软件——日志是 AI 的眼睛，闭环是 AI 的思维，共情是你与 AI 的桥梁。 愿你的每一次调试，都能从“摸黑走路”变成“拨云见日”。
