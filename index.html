嵌入式 · AI 协作范式

AI编程调试的五层进化

从零散的「静态猜测」到「工程化闭环」，每一招都解决上一层的痛点，最终实现最低成本、最高可靠性的 Vibe+Verify 混合范式。
第零层 · 起点

把代码贴给 AI，等它“猜”

典型场景：遇到 bug → 复制代码片段 → 描述现象 → AI 列出 5~6 种可能原因（终端电阻？波特率误差？滤波器？）。开发者逐一排查，运气好半小时，运气差半天毫无进展。

根本痛点：AI 只能基于静态代码做静态推理，而大多数 bug 源于动态行为（状态跃迁、变量运行时值、时序）。缺少运行时“眼睛”，再强的模型也只能盲人摸象。
第一招 · 可观测性

结构化日志 — 让 AI 拥有“眼睛”

将运行时状态用 语义化键值对 + 文件名/行号 输出，AI 可直接解析并定位问题，不再罗列通用原因。

#define LOG(level, fmt, ...) \
    printf("[%s] %s:%d " fmt "\r\n", level, __FILE__, __LINE__, ##__VA_ARGS__)

LOG("INFO", "state=%s epoch=%d", state_to_str(state), epoch);
// 输出: [INFO] can_slave.c:412 state=ONLINE_FIXED epoch=2
语义化：ONLINE_FIXED 比数字6更友好
键值对：正则解析，AI 能提取变量
行号超链接：VSCode 终端一键跳转源码
✨ 解决：AI 不再罗列模糊原因，直接指出“第412行 epoch 不同步”。
⚠️ 遗留：AI 仍需完整工程上下文理解 epoch 同步协议逻辑。
第二招 · 统一上下文

让 AI 理解你的「世界」

把固件源码、上位机脚本、协议文档、典型日志放在统一目录，整体交给 AI。AI 可追踪从 PC 命令到 MCU 状态机的完整数据流。

project/
├── firmware/          # 所有 .c/.h (含 LOG 宏)
├── host/              # Python 测试脚本
├── docs/              # protocol.md, 状态映射表
└── logs/              # 正常/异常运行日志

使用方式："请分析该目录下工程，日志在 logs/failure.txt" → AI 自动检索相关文件，理解 epoch, COMMIT 帧的含义与状态机预期。

✨ 解决：AI 分析脱离设计意图的问题消失。
⚠️ 遗留：每次调试上传整个工程 token 成本较高，且修改代码仍需手动复制粘贴。
第三招 · 多模型协作

写代码与分析分离，成本最优

低成本模型 (侧边栏 Copilot / Continue / minimax) 负责高频代码生成与重构。
高性能模型 (DeepSeek-R1 / GPT-4o) 低频用于深度日志分析、架构审查和根因定位。

  • 生成初始代码 → 低成本 (~$0.05)
  • 运行获得日志 → 打包给高性能模型 → 分析并输出 git diff (~$0.3)
  • 低成本模型根据 diff 修改 → 快速迭代
结果：高频写代码用低价模型，低频深度分析用高价模型，总成本降低约 65%，分析质量飞跃。
第四招 · VSCode 一体化闭环

一键完成「修改 → 烧录 → 观察」

将编译、烧录、串口监视、测试脚本全部集成到 .vscode/tasks.json，配合终端复用，秒级闭环反馈。

{
    "version": "2.0.0",
    "tasks": [
        { "label": "build", "command": "make" },
        { "label": "flash", "command": "openocd ...", "dependsOn": ["build"] },
        { "label": "monitor", "command": "python -m serial.tools.miniterm COM4 115200" },
        { "label": "full_loop", "dependsOn": ["build","flash","monitor"] }
    ]
}

按下 Ctrl+Shift+B → full_loop，自动编译、烧录、打开串口监视器。日志中的 can_slave.c:412 可直接跳转源码，AI 助手 (Continue/Cline) 实时读取终端输出，形成零切换调试闭环。

✨ 解决：工具链割裂，反馈延迟从分钟级降到秒级。
⚠️ 遗留：新项目仍需手动编写 LOG 宏、目录结构、tasks.json — 重复劳动。
第五招 · 终极范式

Vibe + Verify：分层协作，极致经济与可靠

Vibe 层：使用极低成本模型（minimax / DeepSeek-V2 异步 API）以“氛围编程”模式，快速生成已内置结构化日志和标准工程骨架的模板。
Verify 层：用高性能模型分析运行日志，输出精准修改（git diff）。
两层通过 agents.md 统一行为规范，消除认知断层。

⚙️ 模板生成提示词（给 Vibe 模型）

生成 STM32 CAN 自组织地址协议工程，要求：
1. 使用如下 LOG 宏（包含 __FILE__ 与 __LINE__）
2. 在每个状态转换、帧收发处调用 LOG，输出 key=value
3. 目录结构：firmware/ , host/ , docs/ , logs/
4. 自带 .vscode/tasks.json（build/flash/monitor）
5. 一次性输出完整可编译代码

📄 agents.md（统一协作准则）

# 对 Verify 模型
- 收到日志后必须先定位「文件:行号」
- 输出 git diff 格式修改，附带根因
- 若出现数字状态(state=6)必须要求枚举映射，禁止猜测

# 对 Vibe 模型
- 所有代码必须包含 LOG 宏和关键埋点
- 提供 state_to_str() 函数
- 生成 tasks.json 实现一键闭环

📈 闭环演示（5分钟生成 + 15分钟调试）

$ make full_loop
[INFO] main.c:45 epoch=1
[WARN] can_slave.c:415 epoch mismatch local=1 remote=2

→ 将项目+日志 给 DeepSeek-R1
→ 输出 diff:
diff --git a/can_slave.c b/can_slave.c
@@ -410,6 +410,8 @@

     local_epoch = rx_epoch;
+    LOG("INFO", "epoch updated %d->%d", local_epoch-1, local_epoch);
+    reset_commit_guard();
 }

→ 应用 diff → 再次 full_loop → epoch 同步成功
综合效果： 新项目启动 10 分钟，日志覆盖率 >95%，中等 bug 定位 1-2 轮，总 token 成本约 $0.35/项目，代码可维护性达到产品级。

进阶锦囊 · 让成本再降 50%

异步排队生成 — 使用 minimax/DeepSeek 异步 API 批量生成模板变体，价格再降50%

📌 适用边界：嵌入式状态机、协议栈、后端服务、任何需要精确状态跟踪的系统均适用；一次性脚本或极简原型(<200行)无须过度工程。

核心原则 · 人机共舞的工程化闭环

可观测性内建 — 结构化日志 + 行号是 AI 的眼睛
